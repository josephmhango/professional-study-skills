Why a portfolio, and why GitHub Pages?

Why bother with a portfolio?

  • Your CV is often a static PDF sent in emails.
  • A portfolio site can show:
    • Projects with screenshots and links.
    • Code, data, or reports you are proud of.
    • A clear ‚Äústory‚Äù about what you can do.

Employers, supervisors, and collaborators often:

  • Google your name.
  • Look at your GitHub profile.
  • Appreciate a clean, updated site that‚Äôs easy to share.

GitHub Pages lets you:

  • Host a simple portfolio for free.
  • Update it just by committing to a repo.
  • Avoid complex web hosting or servers.

Tools & mental model

The key ingredients

We‚Äôll use four main tools:

  1. GitHub account
    Your online space for repositories.

  2. Repository (repo)
    A folder on GitHub containing files (Markdown, config, images).

  3. Markdown (.md files)
    A simple way to write formatted text using #, *, -, etc.

  4. GitHub Pages + Jekyll

    • GitHub Pages: the hosting service.
    • Jekyll: a static site generator that turns Markdown into HTML and applies a theme.

You write content in Markdown ‚Üí
Jekyll builds a site ‚Üí
GitHub Pages serves it at a URL.

Static websites: the simple web

Static site = a collection of pre-built files:

  • HTML (structure).
  • CSS (styling).
  • Maybe a little JavaScript (interactivity).

No:

  • Databases.
  • Login systems.
  • Complex server code.

GitHub Pages is designed to host exactly this kind of static content.

For a CV/portfolio, this is perfect:

  • Mostly text, links, occasional images.
  • Rarely needs dynamic features.

How GitHub Pages works (high level)

Conceptually:

  1. You push files (Markdown, _config.yml, images) to your repo.
  2. GitHub runs Jekyll for you on their servers (for Jekyll sites).
  3. Jekyll:
    • Reads your Markdown and config.
    • Applies a theme.
    • Produces HTML files.
  4. GitHub Pages serves those HTML files at your Pages URL.

You do not have to:

  • Install Jekyll locally.
  • Run build commands.
  • Configure servers or domain names (unless you want your own custom domain later).

User site vs project site

Two main types of GitHub Pages site:

  • User (or organisation) site
    • Repo must be named: yourusername.github.io
    • URL: https://yourusername.github.io/
    • Great for your main portfolio.
  • Project site
    • Repo can have any name, e.g. cool-project.
    • URL: https://yourusername.github.io/cool-project/
    • Great for specific projects or modules.

In this workshop, we‚Äôll build a user site for your main CV/portfolio.

Prepare your portfolio repo

Step 1 ‚Äî Create or Open your Repository

If you haven‚Äôt created a repository yet, follow these steps. If you have one, skip to the next section, but read the note about naming.

  1. Log in to GitHub and click the + icon in the top-right corner.
  2. Select New repository.

Important: Naming your repository

The name you choose determines your website‚Äôs address (URL).

Option A: The Main Portfolio (User Site)

  • Repository Name: MUST be exactly yourusername.github.io (e.g., jdoe.github.io).

  • Resulting URL: https://yourusername.github.io/

  • Why: This is a special reserved name that tells GitHub ‚Äúthis is my main personal site‚Äù. It lives at the root of your domain.

Option B: A Project Site

  • Repository Name: Any other name (e.g., my-portfolio, cv, lab-reports).

  • Resulting URL: https://yourusername.github.io/my-portfolio/

  • Why: Standard repositories are hosted in a subdirectory (subfolder) matching the repo name.

  1. Public/Private: Select Public (Pages is free for public repos).

  2. Initialize: Check ‚ÄúAdd a README file‚Äù (this creates the first commit so you can start editing immediately).

  3. Click Create repository.

Step 2 ‚Äî Add the config file _config.yml

We will add a configuration file to control your site‚Äôs settings.

Why do we need this file? Jekyll (the site builder) needs instructions on how to build your site. Without this file, Jekyll will use default settings which often won‚Äôt match what you want (e.g., no theme, basic styling). It is the ‚Äúcontrol center‚Äù for your entire website.

In your new repo:

  1. Click ‚ÄúAdd file ‚Üí Create new file‚Äù.
  2. File name: _config.yml
  3. Paste this template:
title: "Your Name"
description: "Short tagline about who you are."
theme: minima

author:
  name: "Your Name"
  email: "you@example.com"

header_pages:
  - index.md

github_username: your-github-user
linkedin_username: your-linkedin-handle
  1. Replace the placeholders with your details.
  2. Scroll down and Commit new file.

Understanding _config.yml (conceptually)

This file is written in YAML (Yet Another Markup Language), which is used for configuration.

What happens if you delete this file?

  • Your site might still load, but it will lose its theme and styling.
  • Social links and navigation menus will disappear.
  • The title in the browser tab will revert to the repository name.

Minimal Requirements

Technically, Jekyll can run without a _config.yml, but for a functional portfolio, you must at least specify: 1. theme: Without this, you get raw HTML with no styling. 2. title: Essential for search engines and browser tabs.

Key settings explained:

  • title ‚Äî appears in the browser tab and site header.
  • theme: minima ‚Äî tells GitHub to use the built-in minima theme (Clean, simple layout, no extra setup).
  • header_pages ‚Äî a list of pages to show in the top navigation bar.
  • github_username ‚Äî the theme uses this to generate social icons in the footer.

The ‚ÄúFront Matter‚Äù (YAML Header)

You noticed the block at the top of the file between the triple dashes (---). This is called Front Matter.

---
layout: home
title: Home
---

Why is this important?

This is not optional decoration. It is how you communicate with the Jekyll build engine.

  1. The Triple Dashes (---): These tell Jekyll ‚ÄúStop! This file contains metadata, process it first.‚Äù

  2. If you remove it: Jekyll treats the file as a plain text file. Your layout (header, footer, CSS) will disappear, and the page will look broken.

What goes in here?

  • layout: Tells Jekyll which HTML template to wrap around your content.
    • layout: home (specific to the Minima theme) adds the title and social links.
    • layout: page is for standard content pages.
    • layout: post is for blog articles.
  • title: Sets the page title tag (what you see in search results).
  • permalink (optional): Manually sets the URL (e.g., /my-cv/).

Key takeaway: Every page in your site needs this header to inherit the site‚Äôs theme and structure.

  2. Look at the top for a message: ‚ÄúGitHub Pages is currently being built from the main branch‚Ä¶‚Äù

  5. Click the link!

Discovering and Applying Themes

One of the most powerful features of Jekyll is the ability to swap the entire look of your site by changing a single line of code.

1. Supported Themes

GitHub Pages has a set of ‚Äúsupported‚Äù themes that are pre-installed. These are the easiest to use as they require no extra plugins.

  • Gallery: View them at https://github.com/pages-themes.

  • How to use: In your _config.yml, simply set:

    theme: jekyll-theme-cayman

    (Replace cayman with slate, architect, minimal, etc.)

2. Community Themes (Remote Themes)

There are thousands of free Jekyll themes created by the community.

  • Discovery:
    • Jamstack Themes
    • Jekyll Themes
  • How to use: These often require the jekyll-remote-theme plugin.

    1. In _config.yml, remove theme: ....

    2. Add:

      plugins:
  - jekyll-remote-theme

remote_theme: "benbalter/retlab"

      (Replace benbalter/retlab with the username/repo of the theme you want).

Creating Your Own Theme (Advanced)

If you are inclined to build something truly unique, you can create your own theme from scratch. You don‚Äôt need to build a ‚Äútheme gem‚Äù; you just need to understand the Jekyll folder structure.

By adding these folders to your repo, you override the default theme:

  1. _layouts/: Create a default.html file here. This is the HTML skeleton of your site.

    <!DOCTYPE html>
<html>
  <head>
    <title>{{ page.title }}</title>
    <link rel="stylesheet" href="/assets/css/style.css">
  </head>
  <body>
    <nav>...</nav>
    {{ content }}
    <footer>...</footer>
  </body>
</html>

    The { content } tag is where your Markdown content will be injected.

  2. _includes/: Small snippets of HTML (like header.html or footer.html) that you can pull into your layouts using {% include header.html %}.

  3. assets/css/: Place your custom CSS files here.

  4. _sass/ (Optional): If you use SASS/SCSS, put your partials here.

Once you create a _layouts/default.html, Jekyll will stop using the theme‚Äôs layout and use yours instead (assuming your Markdown files specify layout: default).
