Git: what it is (and isn’t)

Definitions (first use): A repository (repo) is your project’s history stored in a hidden .git folder. The working directory is the files you edit. The staging area (index) is your “basket” of changes to include in the next snapshot. A commit is a named snapshot. A branch is a movable label pointing at a commit. HEAD is your current branch/commit.

Git is a local version control system. GitHub is a remote hosting service for your repo. You can use Git offline; you push when you want to share.

How Git stores history (first principles)

Git stores content by hash. Files → blobs, folders → trees, commits point to a tree plus parent commits. Commits are immutable snapshots linked together, enabling history and diffs.

First‑use definitions: remotes, publish, push, pull, fetch, PR, fork

  • Remote: a copy of your repo hosted elsewhere (e.g., origin on GitHub).
  • Publish (GitHub Desktop): create the remote on GitHub and push your local commits there for the first time.
  • Push: upload your local commits to a remote branch. After first push, it updates the same branch next time.
  • Fetch: download new commits from the remote; your files don’t change yet.
  • Pull: fetch and then merge the remote branch into your current branch (files update).
  • Pull Request (PR): a request for maintainers to pull your branch into a target branch (e.g., into main). Not the same as git pull.
  • Fork: a server‑side copy of someone else’s repo under your account; you clone your fork locally.

Mnemonic: Push up your work; pull down others’ work; raise a pull request to propose changes into a shared branch.

Install on Windows

  • Install Git for Windows; choose “Use Git from the command line”, enable Credential Manager, keep CRLF→LF on commit.
  • Install GitHub Desktop; sign in; set default editor and shell.
  • Set identity (applies to all repos unless overridden).
where.exe git
git --version
git config --global user.name "Your Name"
git config --global user.email you@example.com

Your first local repository

GitHub Desktop: File → New repository → name hello-git → initialize with README → Create.

History shows the initial commit locally. Use Publish to create origin on GitHub and push.

mkdir hello-git && cd hello-git
git init
echo "# hello-git" > README.md
git add README.md
git commit -m "Initial commit"
# First push sets upstream ("-u")
git remote add origin https://github.com/<you>/hello-git.git
git push -u origin main

Staging vs committing

Edit files. Stage only what belongs together in the next snapshot; then write a clear message and commit. Small, focused commits make history meaningful.

git status
git add README.md notes.txt
git commit -m "Add notes and expand README"

Viewing history & differences

History shows who changed what and when. Diffs show only what changed, not entire files.

git log --oneline --graph --decorate --all
git diff            # working changes vs file on disk
git diff --staged   # staged vs last commit

Ignoring files with .gitignore

Avoid committing build artefacts, caches, and secrets. Add patterns to .gitignore; Git will skip matching files when you stage.

echo "__pycache__/" >> .gitignore
echo ".ipynb_checkpoints/" >> .gitignore
git add .gitignore && git commit -m "Add ignore rules"

Undo, safely

  • Unstage (take out of basket): git restore --staged file
  • Discard local edits: git restore file
  • Amend last commit (unpublished): git commit --amend
  • Revert a published commit: git revert <hash> (creates a new commit that undoes)

Branches: safe parallel work

A branch is a name pointing to a commit. Create a branch for a task, commit on it, then merge back when done. If main didn’t move, the merge fast‑forwards (just moves the pointer). Otherwise Git creates a merge commit to join lines of work.

Create, switch, merge

Create feature/readme-badges, commit, then merge into main.

git switch -c feature/readme-badges
# ...edit...
git add README.md && git commit -m "Add badges"
git switch main
git merge feature/readme-badges

Conflicts happen — resolve intent

When two branches change the same lines, Git marks a conflict. Open the file, keep the correct final text, stage, and commit. Your message should explain the resolution.

# After editing to remove conflict markers
git add conflicted_file.ext
git commit   # completes the merge

Before we collaborate: key terms (first use)

  • Upstream tracking branch: a pairing between your local branch and its remote counterpart (set on first push with -u). Enables simple git push/git pull without extra arguments.
  • Default remote name: origin (can be others).
  • Default branch: often main (formerly master).

Going online: remotes, push, pull, fetch (with Desktop)

Publish (Desktop) = create remote + first push. Afterwards, use Push origin to upload new commits.

Fetch origin downloads new commits without changing files; Pull origin fetches and merges into your current branch.

git remote add origin https://github.com/<you>/hello-git.git
# First push sets upstream
git push -u origin main
# Later
git fetch
git pull   # fetch + merge

Clone vs Fork

  • Clone: make a local copy of a remote repo you can read. You usually push back to the same remote.
  • Fork: make your own remote copy under your account (server‑side). Then clone your fork locally. Used when you cannot push to the original repo.
# Clone
git clone https://github.com/org/repo.git
# Fork (on GitHub UI), then clone your fork
git clone https://github.com/you/repo.git

Pull Request (PR): what and why the name?

A PR is a proposal to merge changes from your branch (source) into a target branch (e.g., main) in a remote repository. You’re asking maintainers to pull your changes. It is not the same as git pull.

  • Anatomy: title + description; source branch → target branch; linked issues; checks (tests); reviewers.
  • Outcomes: comments/review, changes requested, approval, then merge (merge commit / squash / rebase‑merge) according to project rules.

Desktop opens the PR page in your browser; all discussion and review happens there.

Feature workflow with GitHub Desktop (CLI equivalents)

  1. Create branch for your task (Desktop Branch → New; CLI git switch -c feature/x).
  2. Commit locally as you progress (Desktop Changes; CLI git add + git commit).
  3. Push branch (Desktop Push origin; CLI git push -u origin feature/x sets upstream).
  4. Open a Pull Request (Desktop button → GitHub page); describe what/why, link issues (Fixes #123).
  5. Review: peers comment, request changes, or approve.
  6. Merge: often “Squash and merge” to keep one commit per feature; delete branch if project policy says so.

Issues: tracking work

Use Issues to plan tasks and report bugs. Link them to PRs and commits. “Fixes #42” in a commit/PR description closes the issue on merge.

Authentication on Windows

Use HTTPS with Git Credential Manager (default). If required, use SSH keys and add the public key to GitHub.

ssh-keygen -t ed25519 -C "you@example.com"
# Add id_ed25519.pub to GitHub → Settings → SSH and GPG keys
git remote -v

Rescue kit

  • Revert a published commit: git revert <hash>
  • Reset (mixed) to uncommit locally: git reset HEAD^
  • Reset --hard discards work (use with care): git reset --hard <hash>
  • Reflog to find lost commits: git reflog

Common Windows pitfalls

  • Line endings (CRLF/LF): default autocrlf is fine for mixed teams.
  • Large binaries: use Git LFS for big files; avoid committing raw datasets.
  • Multiple Git installs: ensure the one on PATH is Git for Windows (where git).
  • Identity mismatch: keep user.name/user.email consistent with GitHub.

Steady workflow checklist

  1. Pull main before starting.
  2. Create a branch per task.
  3. Commit small, meaningful changes.
  4. Push and open a PR; request review.
  5. Merge, then clean up branches.
  6. Use revert for published mistakes.
